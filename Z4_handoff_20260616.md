いま進行中の作業は2つ:
1. **dev-question導線**（指定IDの問題を直接開く開発用ルート）の追加 ← 実装スクリプト確認済み、実行待ち
2. その後、**meta 15問に固定キー3項目**（`schemaVersion` / `misconceptionId` / `lossCategory`）を追加

Z4を診断エンジンや学習研究装置に**肥大化させない**。あくまで「解答後の理解修復UI」に留める。ただし将来の弱点カルテに繋ぐための**固定キーだけ**を今のうちに入れる。これが確定方針。

## 3. 禁止事項（最優先・違反厳禁）

1. **日本語を含む JSON / TS を PowerShell の `Get-Content` / `Set-Content` で全文書き換えしない。**
   → 過去にこれで `questions_all_700_v1.json` が文字化け・破損した。原因はWindows PowerShellが UTF-8 を システムANSI(CP932) として読み書きしたこと。
   → 日本語データを編集するときは **Node の `readFileSync` / `writeFileSync(..., 'utf8')`**（BOMなし）を使う。バックアップ復旧は `Copy-Item`（バイト単位なので安全）。

2. **`01_locked_datasets` 配下を編集しない。** 確定済みデータ。SHA256確定済み。

3. **`lib/session.ts` / `lib/storage.ts` / `lib/diagnostics.ts` を勝手に変更しない。**（読むのは可）

4. **正解判定・保存ロジック・選択肢シャッフル・option order を変更しない。**
   - 選択肢シャッフルは**廃止確定**。表示順 = options配列順で固定（`buildOptionOrder()` は恒等順 `[0,1,2,3,4]` を返す）。
   - ただしindex変換インフラ（optionOrders等）は残してあり、解答ログの `selectedOriginalIndex` 整合性のため。これも触らない。

5. **バックアップは必ず `.bak` で終わる名前にする**（`.ts.bak` / `.tsx.bak` / `.json.bak`）。`.backup_*.ts` のような **末尾が .ts/.tsx のバックアップは作らない**（TypeScriptがコンパイル対象にしてしまう）。

6. **`★`（star）は公論出版2026年版の値をそのまま使う。変更禁止。**

## 4. Z4 ExplanationCard とは何か

**Z4は「診断」ではなく「修復」。** 弱点カルテ（診断）が「あなたは法令が弱い」と言う側だとすれば、Z4は間違えた直後に「なぜ間違えたか」を理解させる**治療＝学習介入オブジェクト**。

単なる文章解説ではなく、問題の性質に応じて見せ方を変える:
- 今回の失点（lossTitle）/ 短い解説 / 比較表 / 手順表 / 数字カード / 計算ステップ / 施設距離マップ / 流れ図 / 間違いやすい選択肢メモ（optionMemos）

### 表示スタイル（`explanationStyle`）
実装済みレンダラ: `procedure_table` / `comparison_table` / `number_card` / `calculation_step` / `facility_map` / `process_diagram`
型定義のみで未実装（現pilotでは未使用・将来用）: `adaptation_matrix` / `intuition_fix` / `diagram` / `short`

### 接続方式（QuizClient内・確定）
```
{!isExamMode && revealed && (
  explanationMeta ? <ExplanationCard meta={explanationMeta} />
                  : <section>従来解説（explanation + option_details）</section>
)}
```
- metaあり → Z4カード / metaなし → 従来解説にfallback
- 本番モード（模試 isExamMode）では解説を出さない
- `revealed`（解答後）になって初めて表示

### optionMemos のルール（重要）
- 全問・全選択肢に必須ではない。**混乱しやすい選択肢にだけ補助的に**置く。
- **正解選択肢に試験の答えを直接書かない。** 特に「誤っているものはどれか」問題では、正解選択肢=記述として誤っているもの、他=記述として正しいが試験では選ばないもの。
- 現状の運用パターン: 13/15問は「**4つの誤答(distractor)にメモ、正解にはメモしない**」。

## 6. 確定方針: meta に追加する固定キー3項目

**採用（今入れる）**: `schemaVersion` / `misconceptionId` / `lossCategory`
**保留（今は入れない）**: attempt log / 自信度UI / Z4表示ログ / カード効果測定 / cognitiveOperation / conceptTags / remediationGoal / reviewAction

### なぜ今か
`lossTitle` は表示文で、文言は将来変わる（集計キーとして壊れる）。だから表示文とは別に固定IDを持たせる。meta が15問のうちに入れるのが最も安い（700に増えてからの後付けはコスト大）。

### 設計の縛り（重要）
- **`lossCategory` は型でenum化**し、規定外の値はTypeScriptが弾くようにする（ドリフトをコンパイルエラーで止める唯一の方法）。
- **`misconceptionId` は「1問1ID」ではなく「多問→1誤解」**。同じ誤解を突く複数問は**同じIDを共有**させる（これが将来の集計の価値の源泉）。
- `lossTitle` は今まで通り**画面表示用**。`misconceptionId` / `lossCategory` はユーザーに見せない。
- **3項目とも必須**にして15問を一括で埋める。
- 変更対象は **meta側のみ**: `explanation_meta_types.ts` と pilot 3ファイル。`questions_all_700_v1.json` / lib / locked は**触らない**（hashで不変を確認）。

## 8. dev-question導線（STEP 1の内容）

指定IDの問題を直接開く開発用ルート。`QuizClient.tsx` のみ変更。JSON / meta / lib / locked は触らない。

- URL例: `http://localhost:3000/quiz?mode=dev-question&id=LAW-011-001`
- 仕組み: `buildSessionFromMode` と `sessionMatchesMode` に `dev-question` モードを追加し、`searchParams` から `id` を読み、`buildLastSession({ sessionType:'daily-10', count:1, questionIds:[id] })` で1問セッションを作る。
- `buildLastSession` は `questionIds` を尊重し**指定の1問だけ**を出すことを確認済み。

### 実装上の必須ガード（過去のバグ対策）
`if (mode === 'mock-exam')` は **buildSessionFromMode と sessionMatchesMode の2箇所に同一文字列で存在**する。単純な文字列 `.replace` は先頭1件しか置換せず**2つ目のブロックを誤った関数に挿入する**（→ `Cannot find name 'session'`・戻り値型不一致でtypecheck失敗）。

対策（確定済みスクリプトに実装済み）:
- 各 `.replace` のアンカーを**各関数で一意な「mockブロック丸ごと」の正規表現**にする（一方は `return buildLastSession(...)`、他方は `return session.sessionType === ...`）。
- `replaceExactlyOnce`（マッチ数が1でなければthrow）で誤挿入を物理的に防止。
- 正規表現は `\r?\n` で **LF/CRLF混在に対応**（このファイルは混在している）。
- patch後アサート: `mode === 'dev-question'` が**ちょうど2回**出現すること、2つ目が `sessionMatchesMode` 関数内にあること。
- 編集はNode、書き込みは `utf8`(BOMなし)。データ4ファイルのhash不変を確認。typecheckゲート。

### 既知の副作用（要把握・今は割り切り）
dev-questionは `sessionType: 'daily-10'` でセッションを作るため、**テスト中にわざと誤答すると、その誤答が実際の弱点データ（wrongQuestionIds / weaknessRecords）に入る**。記録は questionId 単位で行われ、セッションのlabel(`DEV: ...`)を見ないため。
- 対策: 確認が済んだら localStorage をクリアして弱点データをリセット。
- 「記録をスキップするガード」は保存ロジックに触れるので**今はやらない**（禁止事項3に抵触）。

## 10. 普遍原則（設計の芯）

- **安定キーと可変表現を必ず分離する。** `lossTitle`(表示) と `misconceptionId`(キー) はその一例。問題側の `card_id` でも同じパターンを既に採用。表示文字列を集計キーに使うと必ず後で壊れる。
- **概念のインフレに注意。** Z4を「学習科学の中核装置」に格上げしていく誘惑があるが、北極星は「短期合格ナビ」。固定キーだけ入れて、計測装置の建設は急がない。
- **固定IDを書きっぱなしにしない。** 読み手のいない固定IDは腐る。早い段階で安価な読み手（例: 結果画面の「今回の失点」を `lossCategory` 単位で束ねて表示）を作り、フィールドが正しく埋まっているか検証する。
